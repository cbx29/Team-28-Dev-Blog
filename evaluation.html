Critical Evaluation

User Interface/ User Experience

The user interface is simple and intuitive. Through the MFC it is simple to navigate between modes. Changing the extremity triggers to be blue and with a shadow in the background makes it more aesthetically pleasing.

Functionality

We have fulfilled and surpassed all the requirements given to us, in terms of functionality, as we have made it possible for multiple people to play games at one time by just using hand gestures. It is the first implementation of multiplayer functionality for MotionInput. It provides a means for children in hospitals or anyone going through rehab to do exercise through games while having fun and to connect with other people.

Stability

From our testing of the system, we can conclude that our code is quite stable. There are no major issues, and it generally runs smoothly. However, due to the nature of the program being camera based, there are some instances where stability is affected. For example, the distance from the webcam, the side which each user is on the screen, and when users move too fast can affect how it interprets the gestures. As seen in the tests, the frames per second only decreases by a few when using the multiplayer mode, so it is not a major issue.

Efficiency

Originally, we implemented the multiplayer hand_landmark_detector by comparing hands in the current frame to hands in the previous frame and calculating the distance between them to keep track of the hand's identity (the closest hand). However, this method required us to store a lot of data. After identifying this inefficiency, we redesigned the hand_landmark_detector so that it can correctly identify hands using only information from the current frame.

Compatibility

MotionInput runs across all Windows machines. Our tests also demonstrate that the code runs smoothly in low power machines, however it would run more smoothly on higher powered machines.

Maintainability

As we were working with multiple groups on the project, we had to ensure our code was maintainable. If another future team were to expand our project, they would be able to easily extend it. We created a new flag in the config.json file called multiplayers_mode which is set to True for multiplayer games or false for single player games. This change is then reflected in the hand_landmark_detector which we also modified to have a single player detection mode and a multiplayer detection mode. This means future teams can easily build their work on top of ours regardless of whether they are working on multiplayer games or not.

Project Management

We were all well-coordinated with weekly meetings with our team and TA where we discussed the current progress and tasks for the week that each team member was working on. Our individual strengths complemented each other making it easy for us to split tasks. When one team member was busy, another team member would help them. Each team member had a task at each point. We used a shared Teams folder to keep track of our deliverables including demos and other information. We also had a WhatsApp group chat to schedule meetings and discuss quick matters. We stuck to our Gantt chart to ensure our long-term goals would be completed within the deadline.

Future Work

  • One key feature in our Could-have requirements is to have detection and tracking of 3 or more users. This would give users more freedom when using the software.
  • To improve, we could also generate an energy report that can be sent to Intel.
